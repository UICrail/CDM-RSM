Ontology Specification Draft

Abstract

This ontology allows to place things on the topology. Things may have locations of type spot-, linear-, or area location. For ordering things, this ontology uses the List ontology published by Pieter Pauwels and Walter Terkaj. The List ontology also underpins the ifcOWL ontology by the same authors. For spot locations (and, by extension, for linear locations) on linear elements, it is possible to indicate offsets, quantitatively or qualitvely, in three directions: tangential (projected on the horizontal plane), transversal (orthogonal to tangential, same plane), and vertical.

Localisation: Description back to ToC

Left for future work: 1 - using intrinsic coordinates in a more explicit way (formal references to GeoSPARQL and to ISO 19148, possibly via IFC or DATEX classes). This would entail the replacement of data properties (atIntrinsicCoordinate, startAt, endAt) by object properties. Whether this is worthwhile is a matter for debate. 2 - offsets were lateral in earlier versions (up to RSM 1.2), and are now generalized to tangential and vertical, effectively defining a 3-axis local coordinate system. Explicitly defining a local cartesian coordinate system would be a conceptual simplification and could make use of IFC classes.

The class of all locations that are neither spot, nor linear. Concrete examples: the location of a GSM coverage area (i.e. the parts of topology elements that are in this area); a set of discontinuous or branching linear locations. An area location refers to a set of topology elements, or element parts: no order, no repetition.

Connectionc back to ToC or Class ToC

IRI: https://cdm.ovh/rsm/topology/topo#Connection

Relation between two distinct ports of one (loop case) or two (common case) net elements. This relation is understood as "adjacency", i.e. the related ports are expected to be at the same coordinates or with a minimal gap. A rare exception would be the case of transfer bridges, turntables, or the like.

Listed Net Elementc back to ToC or Class ToC

IRI: https://cdm.ovh/rsm/localisation/localisation#ListedNetElement

Linked list of all net elements (or parts of linear elements) constituting a location. A subclass of OWL List: this is a class from a vocabulary also used by IfcOwl. In the present case, the order of elements in the list is user-defined and has no universal semantics. Repetition is allowed, but should be avoided as it has no meaning in the case of non-linear locations. Linear elements are referred to as a whole. If only part of a linear element is included in the location, use a LinearElementList individual. Orientation of each linear element (or part thereof) is described by startsAt and endsAt properties.
has super-classes
Net Element c, O W L List c
is in range of
net element list op

application directionop back to ToC or Object Property ToC

IRI: https://cdm.ovh/rsm/localisation/localisation#applicationDirection

Allows to express the direction of travel for which the localized entity is relevant. Example: a signal generally applies to one travel direction; track occupancy detection systems may be oriented, etc. Direction of travel is expressed relative to the orientation of the linear element, from port 0 to port 1, or intrinsic coordinate 0 to 1. It is NOT expressed with respect to "normal travel direction in operations".
