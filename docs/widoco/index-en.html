Abstract

Full RSM topology; relations between ports are only available as classes (possibly temporal objects) as well as object properties (possibly with temporal annotations, using RDF+). Syntactic sugar: some properties that are most likely to be used are chained properties, equivalent to derived associations in UML. Temporality to be expressed using an upper ontology, such as DOLCE+DNS Ultralite.

Boundary elementc back to ToC or Class ToC

IRI: https://cdm.ovh/rsm/topology/topo#BoundaryElement

Linear element with length = 0. It still has two distinct port, allowing to express connectivity and navigability. To be used at the borders of networks managed by different entities.
has super-classes

Connectionc back to ToC or Class ToC

IRI: https://cdm.ovh/rsm/topology/topo#Connection

Relation between two distinct ports of one (loop case) or two (common case) net elements. This relation is understood as "adjacency", i.e. the related ports are expected to be at the same coordinates or with a minimal gap. A rare exception would be the case of transfer bridges, turntables, or the like.

connects port op

has nominal metric lengthdp back to ToC or Data Property ToC

IRI: https://cdm.ovh/rsm/adapters/geosparql_adapter/geo_ad#hasNominalMetricLength

This length is assumed to be derived from the chosen Nominal Geometry. The Nominal Metric Length is supposed to be used for conversions between linear referencing methods "absolute" (distance along, e.g. used in CCS/TMS data model) and "interpolation" (intrinsic coordinates, with values 0 to 1, used in RSM and others).
